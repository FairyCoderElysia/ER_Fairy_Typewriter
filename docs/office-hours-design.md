## 1. 项目定位
ER Fairy Typewriter 是一个个人学习用的轻量搜索引擎项目。它不追求 Google 级规模，而是追求完整、可解释、可复盘的搜索引擎学习闭环。

项目首阶段聚焦二次元游戏、游戏人物、游戏资讯，后续再扩展到动漫、coser、虚拟主播、漫展等更宽的二次元领域。

核心学习链路：

```text
爬虫下载
  -> HTML 解析
  -> 文档清洗
  -> SQLite 持久化
  -> 中文/英文分词
  -> 倒排索引
  -> TF-IDF / 字段权重 / boost 排序
  -> 搜索 API
  -> 搜索页面
  -> 调试解释
  -> 搜索评测
```

## 3. 核心决策
### 3.1 主路线
选择“教学自研路线”为主线，专业搜索引擎作为后期对照。

原因：
- 自研 crawler/parser/store/tokenizer/indexer/search/web 能学到完整原理。
- 过早引入 Elasticsearch/OpenSearch 会遮蔽底层实现。
- 轻量本地项目更适合反复实验、重建和调试。

### 3.2 优先纳入
这些能力正式进入近期路线：

- `/debug/search`：解释一次搜索为什么这样排序。
- 搜索评测集：用 Top1、Top3、Zero Result Rate 判断搜索质量。
- 别名词典/领域词典：支持角色别名、作品简称、中英混合查询。
- `SearchIndex` 抽象：为 Redis 和专业搜索引擎对照铺路。
- 索引状态页：展示文档数、token 数、最近抓取时间、失败页面。

### 3.3 暂缓内容
这些内容暂不进入近期开发：

- 大规模真实爬虫。
- 动漫、coser、虚拟主播、漫展全领域同时覆盖。
- Elasticsearch/OpenSearch 主线接入。
- 复杂门户页、资讯流、热点榜、推荐系统。
- 登录、收藏、历史记录、用户系统。

## 4. 推荐架构
### 4.1 模块边界
保持当前模块拆分：

```text
crawler   负责请求网页、限制域名、深度、页数、频率
parser    负责从 HTML 中抽取标题、正文、标签、图片、链接
sources   负责读取受控数据源配置，并转换成 CrawlConfig
domain_terms 负责加载别名/领域词典，并在入库前补全文档字段
store     负责 SQLite 持久化、URL/content_hash/标题相似度去重、文档读取
tokenizer 负责中文/英文分词、停用词、n-gram
indexer   负责倒排索引、TF-IDF、字段权重、排序
search    负责分页、摘要、高亮、面向 Web/API 的结果包装
web       负责 FastAPI 路由、模板、开发接口
```

### 4.2 统一数据流
数据进入链路：

```text
Seed URL / Local Fixture
  -> SmallCrawler
  -> AnimePageParser
  -> auto category / canonical URL / content_hash
  -> domain terms enrichment / source score
  -> SearchDocument
  -> SQLiteDocumentStore
  -> SearchIndex.rebuild()
  -> SearchService.search()
  -> HTML / JSON Response
```

搜索查询链路：

```text
Search Query
  -> tokenizer.tokenize()
  -> SearchIndex.explain()
       -> query tokens
       -> postings / candidates
       -> field matches
       -> TF-IDF score
       -> exact / alias / freshness boost
       -> final score
  -> SearchService pagination and snippets
  -> HTML / JSON Response
```

关键原则：普通搜索和 `/debug/search` 必须共享同一条底层计分路径。解释功能不是另一个搜索引擎，而是同一个搜索引擎的可观察输出。

### 4.3 阶段 4 已完成的垂直字段

阶段 4 不是单纯“多加几个字段”，而是让搜索开始理解二次元领域语义：

- `aliases`：角色名、作品简称、中英混合别名。
- `entity_type`：区分角色、作品、资讯等实体类型。
- `game_title`：把角色和作品关联起来。
- `character_name`：让角色精确命中更稳定。
- `source_score`：对来源质量做轻微业务加分，而不是替代相关性。

这几个字段已经进入 `SearchDocument`、SQLite 存储和索引评分路径。

### 4.4 建议接口
在 Redis 或专业搜索引擎对照之前，先定义最小 `SearchIndex` 接口：

```python
class SearchIndex:
    def rebuild(self, documents: list[SearchDocument]) -> None: ...
    def search(
        self,
        query: str,
        category: str | None,
        limit: int,
        offset: int,
    ) -> tuple[list[tuple[SearchDocument, float]], int]: ...
    def explain(self, query: str, category: str | None = None) -> SearchExplanation: ...
    def stats(self) -> IndexStats: ...
```

配套结构：

- `SearchExplanation`：保存 query、tokens、candidate_count、results、missing_terms。
- `DocumentScoreExplanation`：保存 doc_id、title、field_matches、tfidf_score、boost_score、final_score。
- `FieldMatch`：保存 field、term、tf、idf、field_weight、contribution。
- `IndexStats`：保存 document_count、term_count、posting_count、last_rebuilt_at。

## 7. 前端设计规则
前端定位是“轻量搜索工具 + 二次元学习实验室”，不是资讯门户，也不是营销页面。

### 7.1 信息架构
```text
首页搜索
  -> 搜索结果页
       -> 结果列表
       -> 分类 / 来源 / 时间筛选
       -> 排序解释入口
       -> 无结果建议
  -> 调试解释页 /debug/search
       -> 查询分词
       -> 候选召回
       -> 字段命中
       -> TF-IDF 与 boost 分数
       -> 最终排序
  -> 索引状态页
       -> 文档数
       -> token 数
       -> 最近抓取时间
       -> 失败页面
  -> 学习路线页
       -> 模块职责
       -> 推荐学习顺序
       -> 练习题
```

### 7.2 视觉方向
- 字体：中文正文优先使用 `Noto Sans SC` 或系统中文字体；调试分数、token、代码片段可使用等宽字体。
- 主色：保留粉色品牌色 `#d9488f`，只用于品牌、焦点、当前筛选和关键操作。
- 功能色：结果链接使用蓝色，来源使用绿色，debug 使用蓝色，成功使用绿色，警告使用琥珀色，错误使用红色。
- 背景：搜索和结果页以白色为主，浅色背景只用于首页或状态提示。
- 间距：采用 4/8/12/16/24/32 的间距层级。
- 组件：卡片只用于重复项、调试分数块、状态面板。
- 文案：按钮使用明确动作，例如“搜索”“解释排序”“重建索引”“运行评测”。

### 7.3 状态覆盖
每个关键功能都要考虑这些状态：

| 功能 | Loading | Empty | Error | Success | Partial |
| --- | --- | --- | --- | --- | --- |
| 搜索结果 | 正在搜索 | 换关键词建议 | 查询失败和重试 | 结果、摘要、高亮 | 缺图片/来源仍展示文本 |
| 排序解释 | 正在计算 | token 未命中 | 解释失败但搜索可用 | 分数字段拆解 | 无命中字段记 0 |
| 数据抓取 | 当前源和进度 | 无数据源配置 | 失败 URL 和原因 | 新增/更新文档数 | 部分失败保留成功 |
| 索引状态 | 正在读取 | 无文档 | 存储或索引异常 | 文档数和 token 数 | 索引可用但抓取失败 |
| 搜索评测 | 正在运行 | 无评测集 | 失败用例 | Top1/Top3/零结果率 | 部分查询退化 |

### 7.4 可访问性要求
- 搜索输入框必须有可访问名称，不能只依赖 placeholder。
- 所有按钮和链接都有清晰 focus 状态。
- 键盘 Tab 能按自然顺序访问。
- 移动端点击目标不小于 44px。
- 正文和背景对比度至少达到 WCAG AA 的 4.5:1。
- 结果标题支持 visited 状态。
- 移动端筛选器使用横向滚动或下拉，不要直接隐藏。
- `/debug/search` 桌面端可左右分栏，移动端折叠为按结果展开的解释面板。

## 8. 存储与迁移策略
SQLite 仍然是当前阶段的正确选择，因为它零部署、单文件、适合学习。

迁移顺序：
1. 保持 `documents` 表轻量，先用 `tags`、`aliases` 和垂直字段表达领域语义。
2. 通过 `_ensure_document_columns()` 做简单列迁移，避免引入复杂迁移框架。
3. 已增加 `aliases`、`entity_type`、`game_title`、`character_name`、`source_score`、`content_hash` 字段。
4. 已增加 `crawl_runs` / `crawl_errors` 表。
5. 当前去重顺序：URL/canonical URL、`content_hash`、同分类同来源下标题相似度。
6. 数据明显变大后，再考虑 PostgreSQL 或专业搜索后端。

当前阶段不引入复杂 ORM。直接使用 `sqlite3` 更透明，更适合学习 SQL、表结构和数据流。

## 10. 性能边界
当前不需要提前上分布式。

- 1 到 1,000 篇文档：继续使用内存索引，全量 rebuild 可接受。
- 1,000 到 50,000 篇文档：补增量索引、索引统计、启动耗时日志。
- 50,000 篇以上：开始对比 Redis ZSet、Meilisearch、Typesense。
- 300,000 篇以上：再讨论分片、异步抓取、队列、OpenSearch/Elasticsearch。

现阶段应优先记录：

- rebuild 耗时。
- 搜索耗时。
- token 数。
- posting 数。
- 候选文档数。
- 评测质量指标。
- crawl 运行次数、失败数和失败阶段。
- 去重命中类型：URL、`content_hash` 或标题相似度。

## 11. 风险与防护
- 排序解释和真实排序不一致：通过共享 `explain()` 计分路径解决。
- 新增真实数据后搜索质量下降：通过搜索评测集发现退化。
- 数据源抓取失败但页面无提示：通过 crawl 状态表和索引状态页暴露。
- 自动分类误判：保留手动 `category` 覆盖，并通过 parser 测试校验常见页面。
- 标题相似度误合并：只在同分类、同来源范围内比较，并保持阈值较高。
- 别名词典污染结果：通过评测集和 debug 输出检查 alias 贡献。
- 文档字段不断膨胀：通过 `SearchDocument` 明确字段职责，避免所有信息都塞进 tags。
- 开发接口误暴露：通过本地-only 文档、环境变量开关或 token 限制。

## 12. 技术路径
- 语言：Python。
- Web：FastAPI。
- 模板：Jinja2。
- 存储：SQLite 起步，后续可对照 PostgreSQL。

- 测试：pytest，重点覆盖搜索质量和模块边界。
